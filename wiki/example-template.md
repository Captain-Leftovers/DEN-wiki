# example-template

**What it is**: Den's advanced-feature showcase flake template — a complete starter config demonstrating namespaces, angle-bracket syntax, cross-platform NixOS/Darwin hosts, mutual providers, context-aware aspects, `hasAspect` queries, `meta.adapter` filtering, and CI checks.

**Why it exists**: To give users a single reference project where every intermediate-to-advanced Den concept appears in working code, so they can copy patterns instead of reading abstract documentation.

**Relates to**: [[namespaces]], [[angle-bracket-syntax]], [[mutual-provider]], [[vm-testing]], [[project-structure]], [[den-schema]], [[context-transformation-pipeline]], [[cross-class-configuration]], [[hasAspect]], [[meta.adapter]], [[den.default]], [[batteries]]

---

## Core idea

The example template is not the default starter — it is a **feature gallery**. Where the `default` template gives you one NixOS host and minimal VM testing, the `example` template adds:

- A NixOS host (`igloo`) **and** a Darwin host (`apple`)
- A `den.homes` standalone home for the same user
- A local namespace (`eg`) with reusable VM and desktop aspects
- Angle-bracket imports (`<den/define-user>`, `<den/primary-user>`)
- Mutual providers (host pushes config to user, user pushes config to host)
- Context-aware aspects (`cooper`, `autologin`, `setHost`)
- `hasAspect` structural queries and `meta.adapter` / `oneOfAspects` filtering
- CI checks that verify cross-provider behavior
- `den.lib.nh.denPackages` for per-host `nh` apps

Use it to learn. Use the `default` template to start a real config.

## Dendritic flake pattern

`modules/dendritic.nix` wires Den with `github:vic/flake-file`:

```nix
{ inputs, lib, ... }:
{
  flake-file.inputs.flake-file.url = lib.mkDefault "github:vic/flake-file";
  flake-file.inputs.den.url = lib.mkDefault "github:vic/den";
  imports = [
    (inputs.flake-file.flakeModules.dendritic or { })
    (inputs.den.flakeModules.dendritic or { })
  ];
}
```

And `modules/inputs.nix` declares inputs close to their usage:

```nix
{
  flake-file.inputs = {
    home-manager = { url = "github:nix-community/home-manager"; inputs.nixpkgs.follows = "nixpkgs"; };
    darwin = { url = "github:nix-darwin/nix-darwin"; inputs.nixpkgs.follows = "nixpkgs"; };
  };
}
```

The top-level `flake.nix` is auto-generated; run `nix run .#write-flake` to regenerate it after changing inputs.

## Cross-platform hosts

`modules/den.nix` declares one user across three targets:

```nix
{ lib, ... }:
{
  den.hosts.x86_64-linux.igloo.users.alice = { };
  den.hosts.aarch64-darwin.apple.users.alice = { };
  den.homes.x86_64-linux.alice = { };

  # enable hm for all users
  den.schema.user.classes = lib.mkDefault [ "homeManager" ];
}
```

The same `den.aspects.alice` (auto-generated from these declarations) feeds into:
- `nixosConfigurations.igloo` — NixOS system + host-bound HM
- `darwinConfigurations.apple` — macOS system + host-bound HM
- `homeConfigurations.alice` — standalone HM, no host context

## Namespaces

`modules/namespace.nix` creates the local `eg` namespace:

```nix
{ inputs, den, ... }:
{
  imports = [ (inputs.den.namespace "eg" true) ];
  _module.args.__findFile = den.lib.__findFile;
}
```

- `inputs.den.namespace "eg" true` → registers `den.ful.eg` and exposes it as `flake.denful.eg`
- `__findFile` → enables the `<...>` angle-bracket syntax inside this flake

After this, any module can use `{ eg, ... }:` as an argument and access `eg.vm`, `eg.xfce-desktop`, etc.

## Namespace aspects

`modules/aspects/eg/vm.nix` shows nested `provides`:

```nix
{ eg, ... }:
{
  eg.vm.provides = {
    gui.includes = [
      eg.vm
      eg.vm-bootable.provides.gui
      eg.xfce-desktop
    ];
    tui.includes = [
      eg.vm
      eg.vm-bootable.provides.tui
    ];
  };
}
```

- `eg.vm` is a reusable aspect (placeholder in this template)
- `eg.vm-bootable.provides.gui` and `.tui` are **sub-aspects** exposed via `provides`
- `eg.vm.provides.gui` and `.tui` are pre-baked bundles you can include in a host

`modules/aspects/eg/vm-bootable.nix` uses NixOS installer CD modules:

```nix
let
  installer = variant: {
    nixos = { modulesPath, ... }: {
      imports = [ (modulesPath + "/installer/cd-dvd/installation-cd-${variant}.nix") ];
    };
  };
in
{
  eg.vm-bootable.provides = {
    tui = installer "minimal";
    gui = installer "graphical-base";
  };
}
```

This is the pattern for building **tested feature bundles** that compose into bootable images.

## Angle brackets

`modules/aspects/defaults.nix` uses `<...>` shorthand:

```nix
{
  den.default.includes = [
    <den/mutual-provider>
    <den/hostname>
    <den/define-user>
    (if config ? _module.args.CI then <eg/ci-no-boot> else { })
  ];
}
```

`<name>` is syntactic sugar for aspect lookup. It only works when `__findFile` is in scope (set by `namespace.nix`).

`alice.nix` also uses batteries this way:

```nix
den.aspects.alice.includes = [
  <den/primary-user>          # alice is admin always
  (<den/user-shell> "fish")   # default user shell
];
```

## Global defaults

`modules/aspects/defaults.nix` sets `den.default` — the aspect applied to every host, user, and home:

```nix
den.default = {
  darwin.system.stateVersion = 6;
  nixos.system.stateVersion = "25.05";
  homeManager.home.stateVersion = "25.05";
};
```

It also demonstrates `den.lib.perHost`, `perUser`, and `perHome` helpers in comments:

```nix
# den.lib.perHost ({ host }: { nixos.foo = [ 42 ]; })
# den.lib.perUser ({ host, user }: { nixos.foo = [ 42 ]; })
# den.lib.perHome ({ home }: { homeManager.foo = [ 42 ]; })
```

These wrappers scope a function to **only** the `{host}`, `{host,user}`, or `{home}` context, preventing accidental duplication when a parametric function appears in `den.default.includes`.

## Context-aware aspects

`modules/aspects/alice.nix` defines `cooper` and `setHost` — aspects that emit config whenever their required context is present:

```nix
den.aspects.cooper = { user, ... }: {
  nixos.users.users.${user.userName}.description = "Alice Cooper";
};

den.aspects.setHost = { host, ... }: {
  networking.hostName = host.hostName;
};
```

`cooper` fires at every `{host,user}` context stage, setting the user's description. `setHost` fires at every `{host}` stage, setting the hostname.

`modules/aspects/eg/autologin.nix` is another context-aware aspect:

```nix
eg.autologin = { user, ... }: {
  nixos = { config, lib, ... }:
    lib.mkIf config.services.displayManager.enable {
      services.displayManager.autoLogin.enable = true;
      services.displayManager.autoLogin.user = user.userName;
    };
};
```

It only activates when the host has a display manager enabled, and only when a user context exists.

## Mutual providers

`modules/aspects/alice.nix` — user pushes config **to** the host:

```nix
den.aspects.alice = {
  provides.igloo = { host, ... }: {
    nixos.programs.nh.enable = true;
  };
};
```

`modules/aspects/igloo.nix` — host pushes config **to** the user:

```nix
den.aspects.igloo = {
  provides.alice = { user, ... }: {
    homeManager.programs.tmux.enable = user.name == "alice";
  };
};
```

During the `ctx.user` stage, `mutual-provider` scans these `.provides.*` entries and forwards them with correct class routing (`nixos` → host module, `homeManager` → user's home).

## hasAspect and meta.adapter

`modules/aspects/hasAspect-examples.nix` contains two complementary patterns:

**Pattern 1 — Reading structure via `host.hasAspect` from a class body**

```nix
den.aspects.example-impermanence = { host, ... }: {
  nixos = { lib, ... }:
    lib.mkMerge [
      (lib.mkIf (host.hasAspect den.aspects.example-zfs-root) {
        environment.etc."example-impermanence-flavor".text = "zfs";
      })
      (lib.mkIf (host.hasAspect den.aspects.example-btrfs-root) {
        environment.etc."example-impermanence-flavor".text = "btrfs";
      })
    ];
};
```

`host.hasAspect` queries the **resolved** aspect tree at evalModules time. This is cycle-safe because the tree is already frozen.

**Pattern 2 — Deciding structure via `meta.adapter` (`oneOfAspects`)**

```nix
den.aspects.example-secrets-bundle = {
  includes = [
    den.aspects.example-agenix-rekey
    den.aspects.example-sops-nix
  ];
  meta.adapter = den.lib.aspects.adapters.oneOfAspects [
    den.aspects.example-agenix-rekey
    den.aspects.example-sops-nix
  ];
};
```

`oneOfAspects` is a `meta.adapter` that keeps the first present aspect and tombstones the rest. It runs **during resolution**, operating on the tree rather than from inside it — so it cannot cycle.

**Anti-pattern**

Using `hasAspect` inside an `includes` list causes infinite recursion:

```nix
# BROKEN — do not do this
den.aspects.broken = { host, ... }: {
  includes = if host.hasAspect den.aspects.example-zfs-root
    then [ den.aspects.example-zfs-impermanence ]
    else [ den.aspects.example-btrfs-impermanence ];
};
```

Why: `resolved` depends on `includes`, and `includes` depends on `resolved`. A classic fixed-point with no fixed point.

See [[hasAspect]] and [[meta.adapter]] for full details.

## Tests

`modules/tests.nix` verifies the cross-provider wiring and custom emacs:

```nix
checks."alice enabled igloo nh" =
  checkCond "alice.provides.igloo" igloo.programs.nh.enable;

checks."igloo enabled alice tmux" =
  checkCond "igloo.provides.alice" alice-at-igloo.programs.tmux.enable;

checks."alice-custom-emacs" =
  checkCond "hm.programs.emacs.package" (
    "emacs-nox" == lib.getName alice-at-igloo.programs.emacs.package
  );
```

These are flake checks that run during `nix flake check` on Ubuntu and macOS (see `.github/workflows/test.yml`). They prove the mutual providers actually fired and the custom emacs package is set.

## VM

`modules/vm.nix` injects VM-specific aspects into the igloo host:

```nix
den.aspects.igloo.includes = [
  eg.vm.provides.gui
];
```

And wraps the VM build in a shell app:

```nix
perSystem = { pkgs, ... }: {
  packages.vm = pkgs.writeShellApplication {
    name = "vm";
    text = ''
      ${inputs.self.nixosConfigurations.igloo.config.system.build.vm}/bin/run-igloo-vm "$@"
    '';
  };
};
```

Run `nix run .#vm` to launch the QEMU VM.

## Data flow

```
den.nix (igloo/apple + alice)
        │
        ▼
   ctx.host ──► aspects.igloo (nixos + provides.alice)
        │
        ├──► ctx.user ──► aspects.alice (includes + provides.igloo)
        │                    │  ▲
        │                    └──┘ (mutual routes)
        │
        └──► ctx.hm-host ──► nixosConfigurations.igloo
                              darwinConfigurations.apple
                              homeConfigurations.alice
```

Both `aspects.igloo` and `aspects.alice` contribute to the final system. The mutual providers create a loop: each can read the other's resolved config.

## Gotchas

- **The `eg` namespace and `eg/` folder are coupled for teaching clarity** — not a structural recommendation. In real configs, the namespace name (`my`) should not match a folder name. See [[project-structure]] for why.
- **VM settings are injected into the host aspect** — `eg.vm.provides.gui` is included in `den.aspects.igloo.includes`. This means VM-specific config travels with the host even on real hardware. Harmless for `eg`'s minimal settings, but a bad pattern for real configs. The `default` template keeps VM settings out of the host aspect.
- **Darwin hosts cannot use `system.build.vm`** — the VM runner only works for NixOS hosts. The `apple` host has no VM package.
- **Standalone homes are not tested by the VM** — `den.homes.x86_64-linux.alice` produces a separate `homeConfigurations.alice` output. Test it with `home-manager build --flake .#alice`.
- **hasAspect is not for includes** — querying resolved state from inside `includes` causes infinite recursion. Use `meta.adapter` (e.g. `oneOfAspects`) for structural decisions.
- **perHost / perUser / perHome in defaults** — parametric functions in `den.default.includes` affect every host, user, and home. If a function appends to a list option without scoping, it duplicates. Use `den.lib.perHost` / `perUser` / `perHome` to restrict activation.

## In the Nix ecosystem

The example template sits between "hello world" and "production flake". It is Den's equivalent of a framework's "example app" — every feature is wired together so you can see interactions that isolated docs cannot show.

Compare with:
- `default` template — minimal, one NixOS host, no namespaces, no angle brackets
- `ci-tests` template — exhaustive feature coverage, not meant for human reading
- `nvf-standalone` template — library-only usage, no hosts or users at all

If you are migrating an existing config and wondering whether to adopt namespaces or mutual providers, study this template first, then decide which pieces to port.
